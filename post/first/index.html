配置

  • 配置名称: .gitconfig
  • 配置路径：
    1. windows: C:\Users\<用户名>\.gitconfig
    2. macos: /Users/<用户名>/.gitconfig
    3. linux: ~/.gitconfig

配置级别

仓库级别

local 【优先级最高】 local 的配置存放在当前仓库下，即：.git/config 下。

可通过命令行 git config --local core.bare false ，即：

# 编辑配置
# 当然，如果不知道本地配置的路径，可以使用这个函数查看。如果没有文件，会创建文件。
git config --local -e

# 查看仓库本地配置
git config --local -l

# 设置仓库本地配置
git config --local <moudle>.<property> <value>

# 移除配置
git config --unset --local <moudle>.<property>

用户级别

global 【优先级次之】 global 的配置存放在用户目录下，即：~/.gitconfig下。

# 编辑配置
git config --global -e

# 查看仓库本地配置
git config --global -l

# 设置仓库本地配置
git config --global <moudle>.<property> <value>

# 移除配置
git config --unset --global <moudle>.<property>

系统级别

system 【优先级最低】 system 的配置存放在 /etc/gitconfig 下。

# 编辑配置
# 可以使用此命令查看配置文件的路径
git config --system -e

# 查看仓库本地配置
git config --system -l

# 设置仓库本地配置
git config --system <moudle>.<property> <value>

# 移除配置
git config --unset --system <moudle>.<property>

存储分类

其实工作区可以包含未跟踪区，只是未跟踪区的文件无法被stash，所以分开展示。

  • 未跟踪区： 只包含新添加的，未跟踪的文件。
  • 工作区： 包含删除的文件，修改的文件。
  • 暂存区： 像未跟踪文件、删除的文件、修改的文件经过add后，都会进入暂存区。
  • 临时区： 临时区保存了工作区的修改状态。
  • 本地分支： 暂存区提交后进入本地分支。
  • 远程分支： 本地分支push后进入远程分支，一般远程分支在服务器上。

未跟踪区<=>暂存

# 添加暂存
# 将【未跟踪的文件】添加到暂存区
git add <file>

# 取消暂存
# 回退添加到暂存区的文件
git restore --staged <file>

工作区=>暂存区

# 添加暂存
# 将【删除的文件、修改的文件】添加到暂存区
git add <file>

# 取消暂存
git restore --staged <file>

暂存区<=>本地分支

# 提交
git cm -m "brief meesage"

# 取消提交
# <number> 代表上<number>个版本
# --soft 不会删除工作区的改动。只是撤销前面的commit，并且回到暂存区。
# 可以指定<file>，如果不指定，回退所有的提交。
git reset --soft HEAD~<number> [<file>]

# --hard 会删除工作区的改动，且撤销commit，直接回到工作区（也就是会撤销add）
git reset --hard HEAD~<number>
# 撤回到指定commit提交前
git reset --hard <commit-id>
# 撤销到和远程一样
git reset --hard origin/master

工作区<=>临时区

# 添加临时区
git stash

# 取消临时区
# 将临时区内容恢复到工作区，并删除暂存区内容
git stash pop

# 撤销修改，但不更改存储区类型。即：原来在哪个存储区就在哪。
# 	1. 当文件在工作区，则撤销当前文件修改。
# 	2. 当文件在暂存区，且在工作区对此文件又做了修改，则撤销到暂存区文件的状态。
# 也就是不管在哪个区，都会撤销之前对文件做的更改。

# git reset 相关命令 可查看【暂存区<=>本地分支】介绍

临时存储区

工作区内容临时存储

# 工作区所有内容暂存临时区。git st：工作区是干净的

# 查看当前的暂存列表。从【stash@{0}】开始
git stash list

# 恢复临时区栈顶内容到工作区，并且删除临时区栈顶内容
git stash pop

# 恢复临时区栈顶内容到工作区，并且不删除临时区栈顶内容
git stash apply
# 恢复指定的临时区内容
git stash apply stash@{<number>}

# 删除临时区【队头】内容，即：【stash@{0}】
git stash drop
# 删除指定的临时区内容
git stash drop stash@{<number>}

# 清除临时区全部内容
git stash clear

# 	git co <tag-name> 时候
# 	git会提醒处于"detached HEAD"状态。因为tag是一个快照，所以无法修改其中代码，如果需要修改应当建立分支

分支管理

# 抓取远程分支。只更新到本地对应的分支
git fetch origin [<remote-branch-name>]:[<local-branch-name>]

# 拉取分支，其实 pull 就是 fetch 和 merge FETCH_HEAD 的简写
git pull origin [<remote-branch-name>]:[<local-branch-name>]

# 删除远程分支
git push origin :<remote-branch-name>
git push --delete origin <remote-branch-name>

# 删除本地分支
git br -D <local-branch-name>

# 切换到指定分支
# origin 为远程仓库别名
# 	remotes/origin/master
# 	remotes/origin/release/1.0
# 切换到1.0
git co release/1.0

# 重命名分支
git br -m <old-branch-name> <new-branch-name>

# 设置【upstream】分支
# 当本地初始化，手动关联远程库时，或者检出新的分支需要推送到远程对应分支上时。
# 第一次推送会报错：[fatal: 当前分支 master 没有对应的上游分支。]，此时需要为这个分支配置上游分支。
# 此后，本地 <local-branch-name> 关联到远程同名的分支上
git push -u <remote-host-name> <local-branch-name>

pull 和 fetch 区别

# 要理解 pull 和 fetch 区别，必须理解一个前置概念：跟踪远程分支
# 在项目目录下 .git/refs 保留 【heads】【remotes】【tags】的状态
# 比如：当前更新了远程 master 分支的状态，本地 *master 分支上的 commit-id = 0。
# 那么：.git/refs/heads/master 保存的 hash 即：commit-id = 0。
# 且：.git/refs/remotes/origin/master 保存的 commit-id = 0。
# 此时远程服仓库，其他玩家更新了 master 使得 commit-id = 1

# 1. 如果使用 pull
# 最终会更新 .git/refs/heads/master commit-id = 1 和 .git/refs/remotes/origin/master commit-id = 1
# 也就是它不仅更新了跟踪的远程分支，并且把更新下来的内容应用到了本地的分支上。也就是当前HEAD的指向也被更新了。
git pull origin master

# 2. 如果使用 fetch
# 第一步只会更新 .git/refs/remotes/origin/master commit-id = 1
# 此时的 .git/refs/heads/master commit-id = 0 还没有变化
# 当用户执行 git mg origin/master 才会更改 .git/refs/heads/master
git fetch origin # 会更新所有远程的分支，最新的信息
git fetch origin master # 更新指定 master 分支
git mg orgin/master

rebase

减少分叉

# 设：当 master 分支从 Merge-C0 开始，连续更新了 Merge-C1、Merge-C2。
#    用户 A 从 Merge-C2 引出一个分支 dev，并且在本地更新了2个版本：A-Merge-C3、A-Merge-C4。
# 	 用户 B 向远程仓库 master 分支连续提交了 3 个版本：B-Merge-C5、B-Merge-C6、B-Merge-C7。
# 此时用户 A 如果想将本地的 dev 分支合并到远程 master 上，必须：
git co master
git pull origin master
git mg dev
git push origin master
# 这样会在合并 dev 和 远程 master 的过程中，产生新的 A-Merge-C8 ，整个图谱也会出现分叉

  • 变基图谱
# 设：当 master 分支从 Rebase-C0 开始，连续更新了 Rebase-C1、Rebase-C2。
#    用户 A 从 Rebase-C2 引出一个分支 dev，并且在本地更新了2个版本：A-Rebase-C3、A-Rebase-C4。
# 	 用户 B 向远程仓库 master 分支连续提交了 3 个版本：B-Rebase-C5、B-Rebase-C6、B-Rebase-C7。
# 此时用户 A 通过变基的方式将本地 dev 代码推到远程 master 上：

# 1. 从 dev 切换到 master，更新 master 到最新
git co master
git pull master
# 2. 切换到将要【被合并的分支】dev，执行变基操作，即：变基【合并的分支】master
# 	 可以理解为：使得 master 成为 dev 的基石
git co dev
git rebase master
# 如果有冲突，解决冲突
git add .
# 继续变基，还有可能继续冲突，因为它将 dev 的多个commit，一个个合进 master的
git rebase --continue

# 3. 切换到 master， 合并 dev，并推送到远程仓库
git co master
git mg dev
git push

# 可以看出，变基过程中，先更新了 master，这会使得 B 提交的3个更新，在 A 之前。

cherry-pick

目的：就是将指定的提交（commit）应用于其他分支。

# 将提交 <commit-id> 应用到当前分支
# 这会在当前分支产生一个新的提交
git cherry-pick <commit-id>
